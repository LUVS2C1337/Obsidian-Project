# 🔄 敏捷 vs 传统（瀑布）对比

> PMP 考试敏捷内容占比约 **50%**！本篇系统对比。

---

## 核心理念对比

| 维度 | 传统（瀑布） | 敏捷 |
|------|------------|------|
| **需求** | 前期固定 | 渐进演化 |
| **变更** | 严格变更控制 | 拥抱变更 |
| **交付** | 一次性交付 | 增量迭代交付 |
| **团队** | 职能层级 | 自组织跨职能团队 |
| **客户参与** | 里程碑节点 | 持续参与 |
| **成功标准** | 按计划完成 | 交付客户价值 |
| **项目规模** | 大型、可预测 | 小型、复杂度高 |
| **风险** | 前期识别 | 迭代中持续识别 |

---

## 生命周期选择

```
                   不确定性/变更频率
                        │ 高
      ┌─────────────────┼─────────────────┐
      │  预测型(瀑布)     │    迭代型        │   增量型         │  敏捷型
      │  需求明确         │   需求渐近明细    │   需求明确         │   需求高频变更
      │  一次交付         │   多轮迭代无增量  │   增量交付         │   增量+迭代
      │  如：建筑项目      │   如：研究项目   │   如：版本发布      │   如：软件产品
      └─────────────────┴─────────────────┴─────────────────┘
                        低
```

## Scrum 框架（考试重点）

| 角色 | 职责 |
|------|------|
| **Product Owner** | 管理 Product Backlog，定义价值，优先级排序 |
| **Scrum Master** | 服务型领导，促进 Scrum 流程，移除障碍 |
| **开发团队** | 自组织，3-9 人，跨职能 |

| 事件 | 时间 | 目的 |
|------|------|------|
| Sprint | 2-4 周 | 开发周期 |
| Sprint Planning | ≤8h | 规划本 Sprint 目标 |
| Daily Scrum | 15min | 同步进度，调整计划 |
| Sprint Review | ≤4h | 演示成果，获取反馈 |
| Sprint Retrospective | ≤3h | 回顾流程，持续改进 |

| 工件 | 说明 |
|------|------|
| **Product Backlog** | 所有需求的优先级列表 |
| **Sprint Backlog** | 本 Sprint 要完成的工作 |
| **Increment** | 每个 Sprint 交付的可用产品功能 |

## 敏捷工具

| 工具 | 用途 |
|------|------|
| **看板 (Kanban)** | 可视化工作流，限制进行中工作(WIP) |
| **燃起图/燃尽图** | 跟踪 Sprint 进度 |
| **用户故事 (User Story)** | 格式：As a... I want... So that... |
| **故事点 (Story Points)** | 相对估算，通常用斐波那契数列 |
| **规划扑克 (Planning Poker)** | 团队估算技术，避免锚定效应 |
| **MoSCoW** | 优先级分类：Must/Should/Could/Won't |
| **MVP** | 最简可行产品 |

## ❗ 高频考点

1. **混合方法** — 现在 PMP 考试默认"视情况而定"，不是非此即彼
2. **Servant Leadership** — 敏捷项目经理（Scrum Master）是服务型领导
3. **Tuckman 模型** 在敏捷团队中同样适用
4. **WIP 限制** — 限制在制品数量，减少切换成本
5. **回顾 (Retro)** — 最重要的持续改进事件
6. **DoD (Definition of Done)** — 团队一致认可的完成标准
